# Iron v3.2.0-alpha: Library Authoring Polish

*Released: 2026-05-01*

Iron v3.2 closes ten library-authoring pain points surfaced by the Forge
framework spike during Phase 1. Each fix has a traceable GitHub issue
(#47 through #56) and a locked test fixture; together they unblock
external library authors and make `iron.toml`-based packaging the path
forward. The pure-superset guard against v3.0 holds: every program that
compiled on v3.1 still compiles on v3.2, and the only behavior change at
the parser level (issue #54) is a hard rejection of a syntax that the
v3.0 spec already removed.

## At a glance

- **Distribution (#47)**: fresh `curl ... | sh` installs now ship the
  `lib/diagnostics/` headers, so a hello-world compiles unaided. CI
  exercises the full untar-then-compile contract on Linux and macOS.
- **Library packaging (#48)**: `type = "lib"` emits a real
  `target/lib<name>.a` static archive plus an `iron-stub` of the public
  surface. Consumers add a path-form `[dependencies]` entry and call the
  library's `pub` symbols without vendoring source.
- **Visibility (#52)**: `pub` is now accepted at top level. Cross-module
  references to non-`pub` symbols are rejected with `error[E0320]`.
- **iron.toml version pinning (#51)**: `[package].iron = ">= 3.2.0"` is
  enforced before any compile work; mismatches print a clear migration
  hint.
- **iron run cleanup (#53)**: `iron run` no longer leaves binaries in the
  current working directory. Standalone files write to a tempfile that
  unlinks on exit; package builds write to `target/run/<name>`.
- **String concatenation (#49)**: `String + String` is a built-in, not a
  user-extensible operator. `error[E0202]` was narrowed so it stops
  firing on the legitimate concat case.
- **CLI help (#50, #56)**: a central help registry serves `iron --help`,
  `iron <sub> --help`, and `iron --version` with zero side effects (no
  build, no resolve, no network).
- **Standalone-form removal (#54)**: `func TypeName.method()` is rejected
  universally with `error[E0321]`. The stdlib and 85 integration
  fixtures migrated to `patch object T { func method() { ... } }`.
- **Doc-test CI (#55)**: every fenced ` ```iron ` block in
  `docs/language_definition.md` is type-checked on every push and PR.
  Spec drift is now a CI-blocking failure.

## Per-issue fixes

### #47: Fresh install missing lib/diagnostics

[Issue #47](https://github.com/victorl2/iron-lang/issues/47) -- a fresh
`curl ... | sh` installation could not compile a hello-world because
`~/.iron/lib/diagnostics/` was not staged from the release tarball.
ironc would fail at link time with a missing-symbol error.

The fix lands a CMake `install(...)` rule for `src/diagnostics/` so the
tarball ships `lib/diagnostics/{diagnostics.h,diagnostics.c}`, and a
recurring `install-smoke.yml` workflow runs the full
untar-then-compile-hello-world contract on Linux and macOS for every
push and PR. The workflow includes an explicit regression canary that
deletes `lib/diagnostics/` and asserts the next build fails with the
expected diagnostic, so future drift is caught at the workflow layer
instead of in user reports.

Cross-references:
[install-smoke.sh](../../tests/integration/install-smoke.sh),
[install-smoke.yml](../../.github/workflows/install-smoke.yml).

### #48: type="lib" not supported

[Issue #48](https://github.com/victorl2/iron-lang/issues/48) -- declaring
`type = "lib"` in `iron.toml` previously had no effect: the toolchain
still required a `func main()` and emitted a binary, forcing library
authors to ship a stub `main` and ask consumers to vendor source.

v3.2 implements the full library packaging path. `iron build` for a
`type = "lib"` package emits `target/lib<name>.a` (static archive) and
`target/<name>.iron-stub` (declaration-only `pub` surface, similar in
spirit to a C header). Consumers depend on a library via a path-form
`[dependencies]` entry, and ironc threads `-L<lib-dir>/target -l<name>`
through the link step automatically. `iron init --lib` scaffolds a
ready-to-build library project; `iron run` rejects libraries with a
locked, friendly error.

```toml
# Library author: iron.toml
[package]
name = "mylib"
version = "0.1.0"
iron = ">= 3.2.0"
type = "lib"
```

```toml
# Consumer: iron.toml
[package]
name = "consumer"
version = "0.1.0"
iron = ">= 3.2.0"

[dependencies]
mylib = { path = "../mylib" }
```

Cross-references:
[Project manifest](../language_definition.md#project-manifest),
[lib_consume fixture](../../tests/integration/lib_consume/),
[lib_init_smoke.sh](../../tests/integration/lib_init_smoke.sh).

### #49: String + String concat

[Issue #49](https://github.com/victorl2/iron-lang/issues/49) -- writing
`val s = "Hello, " + name` produced `error[E0202]: operator '+' not
defined for String, String`. The workaround was string interpolation
(`"Hello, {name}"`), which works but is friction at the
language-definition level since the spec already documented `+` as the
concat operator.

v3.2 adds `String + String` as a built-in special case in the
typechecker and HIR lowering. `+` for two strings produces a new string
backed by `iron_string_concat`. The error code `E0202` was narrowed so
it no longer fires on the legitimate concat case; user-extensible
operator overloading remains rejected (a separate concern tracked by
OP-OVERLOAD-EXTEND-01).

```iron
val name = "Alice"
val greeting = "Hello, " + name + "!"  -- "Hello, Alice!"
```

Cross-references:
[Strings](../language_definition.md#strings--unicode-first-immutable-references),
[v3_str_concat_chained fixture](../../tests/integration/v3_str_concat_chained.iron),
[v3_str_describable_example fixture](../../tests/integration/v3_str_describable_example.iron).

### #50: iron --help has side effects

[Issue #50](https://github.com/victorl2/iron-lang/issues/50) -- running
`iron --help` (or any subcommand `--help`) used to walk through full
build setup before printing usage: it would resolve the current
package, fetch dependencies, and emit network requests. On a broken
project or offline machine, `--help` could fail outright.

v3.2 ships a central help registry (`src/cli/help_registry.{c,h}`) and
a pre-dispatch `--help` argv scan. Help output is now produced before
any subcommand body runs. There is no `iron.toml` parse, no dependency
resolution, no compile, and no network access on the help path.
A locked `help_no_side_effects_smoke.sh` fixture asserts the
contract by running every help variant inside a directory with a
deliberately broken `iron.toml`; if any variant exits non-zero, the
smoke fails.

Cross-references:
[Command-line interface](../language_definition.md#command-line-interface),
[help_no_side_effects_smoke.sh](../../tests/integration/help_no_side_effects_smoke.sh).

### #51: iron.toml version pinning

[Issue #51](https://github.com/victorl2/iron-lang/issues/51) -- there
was no way to express "this project requires Iron >= 3.2.0" in
`iron.toml`. A user with an older toolchain could try to build the
project and hit confusing parser errors instead of a clear "upgrade
your compiler" message.

v3.2 adds the `iron` field under `[package]`. The value is a Cargo-style
single-string semver constraint. `iron build`, `iron run`, and
`iron check` enforce the constraint before any other work; mismatches
print a locked friendly error pointing at the install command. The
constraint parser supports the four Cargo-compatible operators
(`>=`, `>`, `=`, caret) plus `^` shorthand.

```toml
[package]
name = "mygame"
version = "0.1.0"
iron = ">= 3.2.0"
```

```text
$ iron build
error: package 'mygame' requires Iron 3.2.0 or newer; this toolchain is 3.1.0-alpha.
hint: re-run install.sh or set IRON_BIN to a newer iron.
```

Cross-references:
[Project manifest](../language_definition.md#project-manifest),
[pin_absent fixture](../../tests/integration/pin_absent/),
[pin_mismatch fixture](../../tests/integration/pin_mismatch/),
[pin_satisfied fixture](../../tests/integration/pin_satisfied/).

### #52: top-level `pub` rejected

[Issue #52](https://github.com/victorl2/iron-lang/issues/52) -- the v3.0
spec calls for `pub` at the top level (`pub func`, `pub object`,
`pub enum`) to mark library public surface, but the parser hard-rejected
it with `expected declaration`. Forge could not mark its public API.

v3.2 accepts `pub` at top level via an accept-and-thread modifier loop
(parser, AST, resolver) and adds a cross-module enforcement gate:
referencing a non-`pub` symbol from a different file emits
`error[E0320]: '<name>' is private to <file>`. Stdlib decls are
carved out via a line-offset gate so the auto-prepended built-ins keep
working without an explicit `pub` annotation.

```iron
pub object Player {
    var health: Int
    var name: String

    pub init(name: String, health: Int) {
        self.name = name
        self.health = health
    }

    pub readonly func describe() -> String {
        return self.name + ": " + self.health.to_string()
    }

    -- private helper: callable only from inside Player
    func recalc_stats() { }
}
```

Cross-references:
[Access Control](../language_definition.md#access-control),
[v3_spec_visibility_example fixture](../../tests/integration/v3_spec_visibility_example/),
[v3_cross_module_private compile_fail](../../tests/compile_fail/v3_cross_module_private.iron),
[v3_pub_top_level_val compile_fail](../../tests/compile_fail/v3_pub_top_level_val.iron).

### #53: iron run cwd-pollution

[Issue #53](https://github.com/victorl2/iron-lang/issues/53) -- `iron run
hello.iron` used to write the produced binary to the current working
directory and leave it behind on exit. Running iron from an editor or
from a project root would litter the repo with build artifacts.

v3.2 makes `iron run` context-aware. For a standalone source file, the
binary is written to a `mkstemp`-allocated path under `$TMPDIR` and
unlinked via an `atexit` hook. For a package build (i.e., the cwd
contains `iron.toml`), the binary is written to
`target/run/<basename>` so it co-exists with `target/` artifacts under
the standard package layout. Either way, the cwd stays clean.

Cross-references:
[Command-line interface](../language_definition.md#command-line-interface),
[run_cwd_clean_smoke.sh](../../tests/integration/run_cwd_clean_smoke.sh).

### #54: standalone form `func TypeName.method()`

[Issue #54](https://github.com/victorl2/iron-lang/issues/54) -- the v3.0
spec removed the standalone method form and required methods to live
inside an `object T { ... }` or `patch object T { ... }` block. The
parser still silently accepted `func TypeName.method()` for stdlib
backwards-compat, which left a hole in the user-facing language.

v3.2 closes the hole. The standalone form is rejected universally with
`error[E0321]: the standalone form 'func TypeName.method()' is removed
in v3.2`, with a locked suggestion pointing at `patch object`. The
stdlib was migrated to `patch object` first (PATCH-01..02), then the
parser-side rejection landed (PATCH-03), then 85 integration fixtures
were bulk-migrated via `scripts/migrate_standalone_to_patch.sh`
(PATCH-04). The §Static/factory paragraph in
`language_definition.md` was rewritten with two positive code blocks
(both compile under the new doc-test CI gate, see #55).

```iron
-- v3.1 and earlier: standalone form (now rejected with E0321)
func Counter.bump() {
    self.value = self.value + 1
}

-- v3.2: patch form
patch object Counter {
    func bump() {
        self.value = self.value + 1
    }
}
```

Cross-references:
[Functions and Methods](../language_definition.md#functions-and-methods),
[v3_standalone_method_form compile_fail](../../tests/compile_fail/v3_standalone_method_form.iron),
[migrate_standalone_to_patch.sh](../../scripts/migrate_standalone_to_patch.sh).

### #55: doc-test CI as merge gate

[Issue #55](https://github.com/victorl2/iron-lang/issues/55) -- the
language definition contained code examples that did not compile
(visibility example, patch describable example, static-method
paragraph). There was no automated check, so spec-vs-implementation
drift accumulated silently between releases.

v3.2 ships `scripts/test_doc_examples.sh`: a bash + awk extractor that
walks `docs/language_definition.md`, pulls every fenced ` ```iron `
block, wraps statement-only blocks in a `func main() { ... }` harness,
and runs `iron check` per block. A paired GitHub Actions workflow
(`doc-test.yml`) runs the extractor on every push and PR against a
freshly-built `./build/iron`. An escape hatch
(` <!-- doctest-skip: <reason> --> `) is supported but unused on v3.2
ship day -- every real-Iron block in the spec compiles cleanly. Drift
fixes for the three outdated examples shipped in earlier phases (VIS-05
in Phase 93, STR-03 in Phase 96, PATCH-04 in Phase 98) so the first
green run was unconditional.

Cross-references:
[scripts/test_doc_examples.sh](../../scripts/test_doc_examples.sh),
[.github/workflows/doc-test.yml](../../.github/workflows/doc-test.yml).

### #56: iron --version not in --help

[Issue #56](https://github.com/victorl2/iron-lang/issues/56) -- `iron
--help` and `ironc --help` did not list `--version`. Users had to know
the flag existed.

v3.2's central help registry (see #50) lists `--version` alongside
every other top-level flag. Both `iron --version` and `iron --help`
now emit the same canonical version string, including build metadata.

Cross-references:
[Command-line interface](../language_definition.md#command-line-interface),
[help_no_side_effects_smoke.sh](../../tests/integration/help_no_side_effects_smoke.sh).

## Forge spike rebuild

Phase 99 Plan 03 ran the v3.2 toolchain against the Forge framework
Phase 1 spike at `~/code/forge` (SHA `42435c5e`). The script
`scripts/rebuild_forge_spike.sh` first detects which #47 / #48 / #52
workarounds are still in place in the spike, removes the mechanical
ones (curl-fetch-diagnostics in the spike's CI workflow; explicit
`type = "bin"` flip if present), logs the content-side ones (`pub`
markers across the spike's `src/` tree are author-owned and not
synthesized blindly), and runs `iron build` against this repo's
`./build/iron`. The trap-based EXIT handler restores the spike to its
original SHA on every exit path so the acceptance run is non-destructive.

Outcome: the live rebuild PASSED with `rc=0` in 5 seconds; the produced
binary at `target/forge` ran with `--help` (`rc=0`). The v3.2
toolchain handles the un-modified Phase 1 spike without any of the
defensive workarounds being needed. The script is a developer-machine
acceptance check, not a recurring CI gate (Forge is a separate repo);
it short-circuits with `rebuild_forge_spike SKIP` and `rc=0` when
`$FORGE_DIR` is absent so CI runners pass cleanly.

See [scripts/rebuild_forge_spike.sh](../../scripts/rebuild_forge_spike.sh).

## Doc-test CI

`scripts/test_doc_examples.sh` was added in Phase 99 Plan 01 (commits
`3d011e50` and `3acf3520`). On v3.2 ship day the extractor walks
`docs/language_definition.md`, finds two real-Iron fenced blocks (the
PATCH-04 patch-form migration example at lines 543-553 and the in-block
method example at lines 558-566), wraps statement-only blocks in
`func main() { ... }` per a six-keyword opener heuristic, runs
`iron check` per block, and exits 0 with `Results: total=2 pass=2
skip=0 fail=0`. Zero `doctest-skip` markers are present in the spec
doc; the first-time green run is the durable lock against future
spec-vs-implementation drift.

The diagnostic format on failure is locked at three pieces (source line
range, wrapped source actually fed to ironc, captured ironc stderr) so
a future drift is reproducible from a single CI log line.

## Install

```bash
curl --proto '=https' --tlsv1.2 -sSfL https://ironlang.dev/install.sh | sh
```

```bash
curl --proto '=https' --tlsv1.2 -sSfL https://ironlang.dev/install.sh | sh -s -- --version v3.2.0-alpha
```
