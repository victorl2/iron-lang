# Iron v2.2.0-alpha — Ergonomics

*Released: 2026-04-20*

Iron v2.2 lands three ergonomic upgrades targeting real-game code:
decimal formatting for UI, idiomatic mutable-receiver syntax, and a
hardened per-stream audio registry with typed failure. The full
milestone spans 4 phases (78, 79, 80, 81) and 21 requirements; the
pure-superset guard held throughout — every `.iron` file that compiled
under v2.1 compiles under v2.2 with identical semantics.

## Formatting — `Int.to_string`, `Float.to_string`, `String.pad_left`

Real-game UI code can now stop hand-rolling decimal conversion:

```iron
-- Before v2.2 — pong.iron had a 15-line `int_to_str` helper
val label = int_to_str(score)  -- hand-rolled

-- v2.2 — stdlib formats the number + fixed-width pad for scoreboards
val label = score.to_string().pad_left(3, " ")
-- "  0", "  5", " 42", "999"
```

`Int.to_string` and `Int32.to_string` handle negatives and zero
end-to-end. `Float.to_string` uses libc `%.6g` with trailing-zero trim
and NaN / ±Inf normalization (platform-stable output across macOS,
Linux, and Windows). `String.pad_left(s, width, fill_char)` and
`String.pad_right(s, width, fill_char)` take a typed `fill_char: String`
so callers write `" "`, `"0"`, etc. idiomatically and produce
fixed-width output.

Full test matrix: `tests/integration/fmt_to_string.iron` (71 lines
exercising FMT-01..04 + FMT-06 edge cases, with `.expected` captured
from the binary — no hand-authored expectations drift).
`examples/pong/pong.iron` migrated to
`score.to_string().pad_left(3, " ")` — real-consumer validation, no
visual regression, and 15 lines of bespoke `int_to_str` helper deleted.

**Requirements:** FMT-01 through FMT-06.

## Mutable receivers — `func (mut t: T) method()`

Type-safe mutation through receiver-method syntax:

```iron
object Counter {
    var value: Int = 0
}

-- v2.2: `mut` on the receiver binds it with mutable capability.
func (mut c: Counter) bump(n: Int) {
    c.value = c.value + n
}

func main() {
    var c = Counter(0)       -- var-binding = mut-callable
    c.bump(3)                -- OK: c is var-bound
    println("{c.value}")     -- 3
}
```

Under the hood the compiler flips mut-receiver methods to pointer-
receiver form (`void Iron_counter_bump(Iron_Counter *_v1, int64_t _v2)`
with call-site `&c`), so mutations persist across the call boundary
without changing the Iron-side call shape. Non-mut receiver-form
methods keep their Phase 79 by-value ABI unchanged — this is a
strict addition, not a retrofit.

The typechecker rejects three common mistakes with source carets:

**MUT-03 (E0234)** — mutating a field on an immutable receiver:

```
func (c: Counter) bump() {
    c.value = c.value + 1  -- error[E0234]: cannot mutate field on
                           --               immutable receiver
                   ^~~~~~~
}
```

**MUT-04 (E0235)** — calling a `mut`-method on a `val`-bound value:

```
val c = Counter(0)
c.bump(3)  -- error[E0235]: cannot call mutable method on immutable
           --               binding
^~~~~~~~~
```

**MUT-09 (E0236)** — `mut` on a primitive receiver:

```
func (mut n: Int) negate() { ... }
-- error[E0236]: 'mut' on non-struct receiver type 'Int' —
--               only struct/composite types support mut
```

The pure-superset guard held: every `.iron` file that compiled
under v2.1 still compiles under v2.2 with identical semantics.
Mass-migration to `mut` is **not** required — opt-in adoption
only.

Test coverage: `tests/integration/mut_receiver_basic.iron` +
`mut_receiver_mixed.iron` (MUT-07 persistence + MUT-08 mixed-capability
same-type) + 4 `tests/compile_fail/mut_*.iron` fixtures locking
E0234 / E0235 / E0236 / E0101 caret messages (the E0236 fixture
double-checks the em-dash U+2014 round-trips correctly in stderr).

**Requirements:** MUT-01 through MUT-09.

## Audio — 16-slot play registry + typed failure

v2.2 hardens the raylib audio surface against silent drops. Playing
a sound now returns a diagnosable result:

```iron
enum AudioError {
    NoFreeSlot,       -- all 16 registry slots are occupied
    DeviceNotReady,   -- !IsAudioDeviceReady(); no audio backend
}

-- Idiomatic consumption: match on the Result
match Sound.play(bounce) {
    Result.Ok(_)  -> { }                   -- successful play
    Result.Err(e) -> { log_miss(e) }       -- typed failure
}

-- Or discard-binding for fire-and-forget effects
val _r = Sound.play(bounce)                -- benign-silence on saturation
```

Stop releases the slot immediately, making it available for the next
`play`; restart reuses the freed slot (may be the same index).
`Audio.detach_all()` walks the registry, stops every active Sound +
Music, and empties all 16 slots — useful in scene transitions and
teardown. `Audio.active_slot_count() -> Int32` exposes registry
occupancy for diagnostics and tests.

```iron
-- Under the cap: every play succeeds
val r = Sound.play(s)
-- r is Result.Ok(_)

-- At saturation: typed failure (NoFreeSlot)
-- ... after 16 concurrent plays ...
val r17 = Sound.play(s17)
-- r17 is Result.Err(AudioError.NoFreeSlot)

-- Mass-stop via detach_all
Audio.detach_all()
-- Audio.active_slot_count() == 0
```

The 16-slot cap matches raylib's `MAX_AUDIO_BUFFER_POOL_CHANNELS` —
we never hand out a slot raylib can't play. The registry is a **new**
structure (distinct from the Phase 68 callback trampoline at
`g_audio_cb`), unified across Sound + Music so
`AudioError.DeviceNotReady` and mixed-workload `detach_all` cleanups
share one 16-entry pool.

Canonical compile-only test: `tests/manual/audio_slot_bookkeeping.iron`
(176 lines) walks all 10 steps of the AUDIO-06 spec — 16 aliases + 16
plays + 17th Err(NoFreeSlot) probe + stop 8 + replay 8 reusing freed
slots + `detach_all` + post-detach replay from slot 0. All 26
`Sound.play` Results bound to named vals; `val _k_*` keep-alive
references prevent DCE elision.

**Requirements:** AUDIO-01 through AUDIO-06.

## Migration — `Sound.play` / `Music.play` signature break

This is a **scoped API break** in v2.2: `Sound.play` and `Music.play`
now return `Result[Void, AudioError]` instead of `Void`. The break is
*soft* in practice — Iron allows discarding function return values at
call sites, so existing `Sound.play(s)` one-liners still compile (the
Result struct is constructed and immediately discarded, with no lint or
warning). Migration is voluntary ergonomics rather than mandatory
breakage repair.

Two idiomatic explicit forms when you do want to consume the Result:

**Match** (when you want to branch on failure — test smoke, diagnostic
logs, loud-failure modes):

```iron
match Sound.play(s) {
    Result.Ok(_)  -> { }
    Result.Err(_) -> { }
}
```

**Discard-binding** (when audible silence is a benign degrade, hot
loops, fire-and-forget effects):

```iron
val _r = Sound.play(bounce)
```

The `_r` convention is required today because ironc rejects
`_ = Sound.play(...)` as `error[E0102]: expected expression` — the
bare `_` token is a match-wildcard, not a valid LHS of an assignment.
This is a deferred ironc improvement (see "What's next" below).

In-tree consumers migrated alongside the signature change:

- `tests/manual/audio_smoke.iron` — uses explicit `match` at both its
  AUDIO-06 and AUDIO-10 sites (visible contract probe).
- `examples/pong/pong.iron` — 4 `Sound.play(bounce)` sites for wall
  reflections + paddle collisions use `val _r` (hot-loop brevity).
- `examples/raylib_showcase/raylib_showcase.iron` — SPACE-blip site
  uses `val _r`.

`Sound.stop` / `Music.stop` continue to return `Void` and remain
idempotent (safe to call on a non-playing sound — the registry
lookup is a safe no-op when the buffer isn't registered).

## Compiler — Void-payload ADT support

First stdlib consumer of ironc's new capability. `Result[Void, E]` is
now a valid generic enum — the per-variant payload emitter in
`src/lir/emit_structs.c` skips fields whose resolved type is
`IRON_TYPE_VOID` and emits a `char _dummy;` placeholder when a variant
ends up field-less (keeps the struct body valid C under `-pedantic`).
Capital `Void` is accepted alongside lowercase `void` in the primitive
type-name table in `src/analyzer/typecheck.c` to match
`iron_type_to_string`'s canonical spelling.

The Iron-side surface is unchanged: user code writes
`enum Result[T, E] { Ok(T), Err(E) }` and `Result.Err(AudioError.NoFreeSlot)`
exactly as before. This opens the door for future stdlib patterns
like `Option[Void]` (equivalent to `Bool`), `Either[Void, T]`, etc.

## What's next — v2.3

- **ASSET-01/02** — asset pipeline (packaging + hot-reload).
- **VENDOR-01** — raylib 6 vendor bump completed in mainline; `Image.load_svg`
  remains blocked because upstream still does not expose it in the public header.
- **RCV-GEN-01/02** — generic receivers (`func (l: List[T]) push(item: T)`).
- **Wildcard-discard statement** — `_ = <expr>` at statement position.
  Today requires `val _r = <expr>` as the idiomatic discard form.
- **Generic-enum completion at return sites** — `func f() -> Result[Void,
  E] { return Result.Err(e) }` today fails to infer `T = Void`; the
  completion helper fires at val-decl but not at RETURN sites. Would
  remove the current val-annotation ceremony for Void-payload Results.
- **RLGL-01**, **VR-01** — low-level graphics + VR stereo.

See `.planning/REQUIREMENTS.md` for the full v2.3+ backlog.
