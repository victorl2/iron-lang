**Test 1: `tests/integration/copy_prop_trivial.iron`**

Tests that copy propagation correctly handles the `val x = y` pattern without changing observable behavior:
```iron
fn main() {
    val a = 10
    val b = a
    val c = b + 5
    println(c)

    val x = 42
    val y = x
    val z = y
    println(z)
}
```

**`tests/integration/copy_prop_trivial.expected`:**
```
15
42
```

The key behavior: copy propagation rewrites the IR so `b` references `a`'s value directly and `z` references `x` directly, but the program output is identical. This tests that copy propagation doesn't break the semantics.

**Test 2: `tests/integration/dce_unused_arithmetic.iron`**

Tests that dead code is correctly eliminated without affecting live code:
```iron
fn main() {
    val x = 10
    val y = 20
    val unused = x + y
    val also_unused = x * y * 3
    val used = x - 3
    println(used)
    println(x + y)
}
```

**`tests/integration/dce_unused_arithmetic.expected`:**
```
7
30
```

The key behavior: `unused` and `also_unused` produce dead values — DCE removes them from the IR. The live values (`used` and the inline `x + y` in println) are preserved. Output must be correct.

**Test 3: `tests/integration/const_fold_arithmetic.iron`**

Tests that constant expressions are evaluated at compile time:
```iron
fn main() {
    val a = 3 + 4
    println(a)

    val b = 10 * 5 - 8
    println(b)

    val c = 100 / 10
    println(c)

    val d = 17 % 5
    println(d)
}
```

**`tests/integration/const_fold_arithmetic.expected`:**
```
7
42
10
2
```

The key behavior: constant folding evaluates `3 + 4` to `7` at IR level, and the emitted C contains the literal rather than the computation. Output must be correct regardless of whether constant folding occurred (clang would fold it anyway), but the optimization makes the IR cleaner.

NOTE on test format: The `.expected` file must have a trailing newline if `println` adds one. Check existing `.expected` files to confirm the convention (println in Iron adds a newline per call).
