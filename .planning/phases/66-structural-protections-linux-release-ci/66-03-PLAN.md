-- **Motivating Incident.** Phase 65 AUDIT-01 ranks 5 and 6 flagged two

```c
/* PROT-04 rewrite (ranks 7 and 8): allocate a fresh Iron_MethodCallExpr or
 * Iron_FieldAccess node instead of reinterpreting Iron_EnumConstruct in
 * place. The old pattern was layout-dependent UB: if Iron_EnumConstruct,
 * Iron_MethodCallExpr, or Iron_FieldAccess grew or reordered fields, the
 * reinterpret would silently corrupt memory. Fresh allocation is
 * structurally safe and costs one arena bump per resolved enum variant. */
if (ec->arg_count > 0) {
    Iron_MethodCallExpr *mc = (Iron_MethodCallExpr *)iron_arena_alloc(
        ctx->arena, sizeof(Iron_MethodCallExpr), _Alignof(Iron_MethodCallExpr));
    mc->span          = ec->span;
    mc->kind          = IRON_NODE_METHOD_CALL;
    mc->resolved_type = NULL;
    mc->object        = (Iron_Node *)ident_node;
    mc->method        = member;
    mc->args          = ec->args;       /* reuse the already-allocated arg array */
    mc->arg_count     = ec->arg_count;
    /* Mutate the original node in place so the AST's upstream parent
     * pointer now sees a METHOD_CALL with the new payload. Minimally
     * invasive: write through a typed pointer to avoid strict-aliasing
     * issues with a direct memcpy. */
    IRON_NODE_ASSERT_KIND(ec, IRON_NODE_ENUM_CONSTRUCT);
    /* Copy the fresh mc layout over ec. Since Iron_MethodCallExpr and
     * Iron_EnumConstruct share the common expression prefix (_Static_assert
     * in ast.h) and Iron_MethodCallExpr is <= sizeof(Iron_EnumConstruct) at
     * compile time, the write fits. Add a _Static_assert to prove it. */
    _Static_assert(sizeof(Iron_MethodCallExpr) <= sizeof(Iron_EnumConstruct),
                   "enum-construct-to-method-call rewrite requires size fit");
    *(Iron_MethodCallExpr *)ec = *mc;
    resolve_expr(ctx, (Iron_Node *)ec);
} else {
    Iron_FieldAccess *fa = (Iron_FieldAccess *)iron_arena_alloc(
        ctx->arena, sizeof(Iron_FieldAccess), _Alignof(Iron_FieldAccess));
    fa->span          = ec->span;
    fa->kind          = IRON_NODE_FIELD_ACCESS;
    fa->resolved_type = NULL;
    fa->object        = (Iron_Node *)ident_node;
    fa->field         = member;
    IRON_NODE_ASSERT_KIND(ec, IRON_NODE_ENUM_CONSTRUCT);
    _Static_assert(sizeof(Iron_FieldAccess) <= sizeof(Iron_EnumConstruct),
                   "enum-construct-to-field-access rewrite requires size fit");
    *(Iron_FieldAccess *)ec = *fa;
    resolve_expr(ctx, (Iron_Node *)ec);
}
```

Design note: The full "fresh allocation with no in-place write" variant would require patching every upstream parent pointer that references ec — an architectural change outside Phase 66 scope. The compromise above allocates a fresh node to get the correct layout, then copies the fresh layout over ec so upstream parent pointers remain valid. The _Static_assert enforces the size constraint at compile time (if a future field is added that breaks it, the build fails immediately — which is exactly what PROT-04 is trying to prevent).

If the executor finds that the _Static_assert fires immediately at current head (e.g., sizeof(Iron_MethodCallExpr) > sizeof(Iron_EnumConstruct)), fall back to Option B: add a new field `Iron_Node *rewrite_target` to Iron_EnumConstruct that holds the fresh replacement, and update every upstream walker to follow rewrite_target if set. Document the choice in the Plan 03 SUMMARY.
