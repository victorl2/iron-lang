CRITICAL — How Log.DEBUG constant fields resolve (from RESEARCH.md Pitfall 3):

The `object Log { val DEBUG: Int }` declares DEBUG as a field. When Iron code writes
`log.DEBUG`, the compiler resolves this as a field access. The C emission for `Math.PI`
(which follows the same pattern in math.iron) must be examined to understand how the
emitted C references the constant value.

Before writing the new log.iron, verify the pattern by checking:
1. How `Math.PI` is emitted in a test program (check generated C for `Math.PI` usage)
2. Whether iron_math.h's `#define IRON_PI 3.14...` is what the compiler emits, or
   whether the compiler emits a struct field access

If the compiler emits `#define IRON_PI` style constants, then iron_log.h needs
`#define Iron_log_DEBUG 0` etc. (no changes to log.iron needed beyond declaring the fields).

If the compiler emits struct field access style, the `object Log {}` with `val` fields
is sufficient and iron_log.h needs no changes (the fields will be zero-initialized).

The executor MUST investigate this before writing new constants — a wrong assumption
leads to garbage values or compilation errors at `log.set_level(log.DEBUG)` call sites.

Name mangling:
  Log.set_level(level) -> Iron_log_set_level(int64_t level)
  Log.DEBUG            -> depends on constant field resolution (investigate above)
