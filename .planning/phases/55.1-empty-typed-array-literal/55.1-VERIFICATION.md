## Deferred Issues (Not Blocking — Pre-existing Gap)

### Interface Arrays as Function Parameters — Codegen Gap

**Candidate requirement:** `IFACE-PARAM-01`

When a function takes an interface array parameter (e.g., `func f(shapes: [Shape]) -> Int { return shapes.len() }`), the C emitter generates the signature with `Iron_SplitList_Iron_Shape` but calls `Iron_List_Iron_Shape_len` inside the body. `Iron_List_Iron_Shape_len` is undeclared — the split-collection interception block handles `.len()` on locally-constructed variables of type `[Shape]` but not on function parameters of that type.

**Reproduction:**
```iron
func count_shapes(shapes: [Shape]) -> Int { return shapes.len() }
```
Error: `error: call to undeclared function 'Iron_List_Iron_Shape_len'`

**Impact on Phase 55.1:** The `empty_literal_call_arg.iron` test was written with `[Int]` instead of `[Shape]` as the parameter type. This exercises the identical type-inference code path (`check_expr_with_expected` at the CALL arg site) without triggering the codegen gap. The workaround is documented in the test file's header comment (lines 8-14) and in 55.1-01-SUMMARY.md under "Deferred Issues."

**This gap predates Phase 55.1 and is independent of the empty-literal type-inference fix.** It should be tracked as a new requirement `IFACE-PARAM-01` for a future phase. Remediation likely touches `src/hir/hir_to_lir.c` and/or the suffix-interception block in `src/lir/emit_c.c`.
