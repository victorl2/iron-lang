# Phase 55.1: Empty Typed Array Literal - Context

**Gathered:** 2026-04-09
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix empty typed array literal `[]` type inference so that a declared annotation propagates into the array-literal check path. The bug site is `src/analyzer/typecheck.c:1827-1888` (`IRON_NODE_ARRAY_LIT` case): when `element_count == 0` the loop at line 1832 never executes, `elem_type` stays NULL, and line 1884 falls through to `IRON_TYPE_ERROR`, producing `[<error>]` as the result type. The declared annotation (`var x: [T] = []`) is never threaded into this code path; it's compared at the var-decl handler after the fact, where `[<error>]` fails the comparison with `E0202: type mismatch expected '[<interface>]', got '[<error>]'`.

Phase 55.1 was isolated from Phase 55 during verification when `var shapes: [Shape] = []; shapes.push(Circle(5))` failed to compile despite the push dispatch fix being complete. This is orthogonal to push dispatch — pure frontend type-inference gap.

**Out of scope (belongs to other phases / future):**
- Cross-statement inference (e.g. inferring `var x = []` from later `x.push(Circle(1))`) — deferred, Iron doesn't do backward bidirectional inference across statements
- Nested empty literals `[[Int]]` inner-empty propagation — deferred, only top-level empty literal is in Phase 55.1 scope (fix at the outer context should cascade, but nested-only test not required)
- Refactoring existing Phase 55 test workarounds (the 2-element initial literals in `push_interface_*.iron`) — not required; those tests still work and prove a different thing (push dispatch on interface split collections, not empty literal support)
- Generic helper that threads expected type into OTHER inference gaps (null coercion, integer widening at call sites, etc.) — future work; Phase 55.1 adds only what the empty-literal fix needs

</domain>

<decisions>
## Implementation Decisions

### Fix approach
- **Threaded expected-type via helper wrapper.** Add a new function `check_expr_with_expected(TypeCtx *ctx, Iron_Node *node, Iron_Type *expected)` in `src/analyzer/typecheck.c`. It handles the empty-literal-with-expected-type case before delegating to the existing `check_expr`:

  ```c
  static Iron_Type *check_expr_with_expected(TypeCtx *ctx, Iron_Node *node, Iron_Type *expected) {
      if (node && node->kind == IRON_NODE_ARRAY_LIT && expected &&
          expected->kind == IRON_TYPE_ARRAY) {
          Iron_ArrayLit *al = (Iron_ArrayLit *)node;
          if (al->element_count == 0) {
              al->resolved_type = expected;
              return expected;
          }
      }
      return check_expr(ctx, node);
  }
  ```

- **Existing `check_expr` signature stays unchanged.** No migration of the ~20-30 existing callers. Only the 4 sites that have an expected type at hand switch to the new helper.
- **Rationale:** With all 4 contexts (var decl, call arg, return, assignment) in scope, localized patches at each site would cost ~60 lines of duplicated pattern. The helper is ~15 lines of implementation + 4 one-line callsite swaps = ~20 lines total. Single source of truth. Future contexts (struct field init, match arm, etc.) can adopt the helper for free.

### Contexts covered
All four contexts where an expected element type is statically known at the literal's use site:

1. **Variable declaration with annotation** — `var shapes: [Shape] = []`. The Phase 55 SC1 unblocker. Non-negotiable. The var-decl handler already knows `vd->annot_type`; pass it into `check_expr_with_expected(ctx, vd->init, vd->annot_type)`.
2. **Function call argument** — `process([])` where the formal parameter is typed `[Shape]`. The call handler iterates arguments and has the parameter type via the callee's function type; pass it in.
3. **Return statement** — `func make_empty() -> [Shape] { return [] }`. The return handler knows the enclosing function's return type; pass it in.
4. **Assignment after declaration** — `shapes = []` where `shapes` already has a resolved type. The assignment handler knows the LHS type; pass it in.

Each callsite is a one-line swap from `check_expr(ctx, x)` to `check_expr_with_expected(ctx, x, expected_type)`.

### Error semantics when annotation is missing
- **`var x = []` (no annotation, no context) → improved error.** Replace the generic `E0202: type mismatch expected '[<interface>]', got '[<error>]'` (which is misleading because there IS no expected type in this path) with a targeted diagnostic:

  > `E0220: cannot infer element type of empty array literal; add a type annotation like \`var x: [T] = []\``

- **New error code:** E0220 (or next available in the sequence — planner checks the existing error code space in `src/analyzer/errors.h` or wherever Iron registers them). Emitted from the ARRAY_LIT case when `element_count == 0` AND no expected type was threaded in.
- **No cross-statement inference.** Iron does not do bidirectional inference across statement boundaries. `var x = []` followed by `x.push(Circle(1))` on the next line still fails at the declaration. The Phase 55.1 fix only activates when the expected type is known at the literal's expression context. This keeps Iron's explicit-type model consistent.

### Test matrix
Positive tests (expected to compile and produce correct output):

- **`empty_interface_var_push.iron`** — `var shapes: [Shape] = []; shapes.push(Circle(5)); shapes.push(Square(3)); println("{shapes.len()}")` → expects `2`. This is the Phase 55 SC1 exact pattern — EMPTY-LIT-02 regression test.
- **`empty_concrete_var_push.iron`** — `var ints: [Int] = []; ints.push(7); ints.push(42); println("{ints.sum()}")` and a parallel `var strs: [String] = []; strs.push("hi")` case → expects `49` and `hi`. Proves the fix works uniformly for primitive and object element types, not just interfaces.
- **`empty_literal_call_arg.iron`** — function taking `[Shape]` parameter called with `process([])` → expects the function to run with an empty shapes array, count = 0.
- **`empty_literal_return.iron`** — `func make_empty() -> [Shape] { return [] }` called in main, result assigned and used → verifies the return-site threading.

Negative test (expected to fail compilation with the improved error):

- **`empty_literal_no_annot.iron`** (or the Iron equivalent for compile-failure tests — planner checks how Iron tests compile failures; if there's a `*.expected-error` convention use it, otherwise document as an inline test with a non-zero expected exit code) — `func main() { var x = [] }` → expects `E0220` error message mentioning "empty array literal" and "type annotation".

**Test location:** `tests/integration/` with `empty_` prefix. Standard `.iron` + `.expected` pair pattern per Phase 54 convention. Auto-discovered by `tests/run_tests.sh`.

## Existing Code Insights

### Reusable Assets
- **`Iron_ArrayLit` struct** — AST node for array literals. Has `element_count`, `elements[]`, `resolved_type`. The fix mutates `resolved_type` directly when a matching expected type is provided.
- **`check_expr` existing recursion** — `check_expr(ctx, node)` is the canonical entry point. The helper wraps it without modifying its signature, so all existing callers stay untouched.
- **`Iron_Type` structures** — `IRON_TYPE_ARRAY` with `element_type` field; `IRON_TYPE_INTERFACE`; `IRON_TYPE_OBJECT`. The helper only needs to check `expected->kind == IRON_TYPE_ARRAY` and overwrite `al->resolved_type = expected`; no type construction or arena allocation required (the expected type is already arena-allocated by the annotation path).
- **`emit_error` + `IRON_ERR_*` codes** — existing diagnostic emission pattern at `src/analyzer/typecheck.c:303`. New E0220 uses the same style.

### Established Patterns
- **Localized fix on the frontend** — Phase 53 (analysis improvements) used similar surgical patches to `value_range.c` and type-related files. Phase 55.1 follows the same style: small, localized, no signature changes to existing functions.
- **Helper functions at the top of `typecheck.c`** — existing file has several static helpers before `check_expr`. New helper goes in the same area.
- **Integration test via `.iron` + `.expected`** — all Iron regression tests use this pattern. Negative tests (compile failures) may need a different pattern; planner checks during research.

### Integration Points
- **Var decl handler** — locates the RHS of a var decl, currently calls `check_expr(ctx, vd->init)`. Swap to helper with `vd->annot_type` as expected.
- **Call expression handler** — iterates arguments via `check_expr(ctx, ce->args[i])`. Swap to helper with the formal parameter type from the callee's function type.
- **Return statement handler** — calls `check_expr(ctx, rs->value)`. Swap to helper with the enclosing function's return type.
- **Assignment handler** — calls `check_expr(ctx, asn->rhs)`. Swap to helper with the LHS resolved type.

Each swap is one line. The planner identifies exact line numbers during planning.

<specifics>
## Specific Ideas

- The helper name `check_expr_with_expected` is intentionally verbose to signal its opt-in nature — callers that pass `expected == NULL` get identical behavior to plain `check_expr`. No magic.
- The fix should NOT try to make `var x = []` work without an annotation. That would require cross-statement inference which is outside Iron's design. Keep the error loud and helpful.
- Once Phase 55.1 lands, Phase 55's regression tests could technically have their 2-element initial literal workarounds simplified to `[]` — but this is NOT required. The existing tests still prove push dispatch correctness (a different thing from empty-literal support). Leaving them as-is is fine.
- EMPTY-LIT-02 regression test name is `empty_interface_var_push.iron` (not `empty_typed_array_push.iron` as originally mentioned in ROADMAP.md SC5 — planner should update the ROADMAP reference to match the actual test filename if they diverge, or pick the name that's cleanest given the other empty_ tests).
- The planner should check if Iron has an existing `*.expected-error` or `*.expected-stderr` convention for compile-failure tests. If not, the negative test can be skipped or implemented as an inline assertion in a wrapper script. Don't invent a new test framework for one negative test.

</specifics>

<deferred>
## Deferred Ideas

- **Cross-statement inference** — `var x = []; x.push(Circle(1))` should not infer `[Circle]` from the push site. Iron's explicit-type model is intentional; changing it is a milestone-level decision, not a Phase 55.1 concern.
- **Nested empty literals** — `var m: [[Int]] = [[]]` (outer non-empty, inner empty) should work naturally once the helper threads through element types of nested arrays — but this is not required by Phase 55.1 and not tested. If it works for free, great; if it needs extra work, defer.
- **Generic expected-type threading for other inference gaps** — null coercion, integer literal widening at call sites (`f(5)` where f takes `i8`), enum variant narrowing, etc. All are candidates for the same helper pattern but out of Phase 55.1 scope.
- **Removing Phase 55 test workarounds** — the 2-element initial literals in `push_interface_*.iron` could be rewritten to use `var shapes: [Shape] = []` once this phase lands. Not required — those tests still cover push dispatch which is what they're meant to prove.
- **`var x: [Shape] = [] as [Shape]`** (explicit cast syntax) — alternative workaround if the helper approach proves problematic during implementation. Only consider if the helper path hits an unexpected blocker.

</deferred>
