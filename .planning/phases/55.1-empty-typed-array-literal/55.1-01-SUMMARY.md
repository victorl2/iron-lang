## Root Cause

`src/analyzer/typecheck.c` IRON_NODE_ARRAY_LIT case (line 1827-1888): the element-type loop at line 1832 iterated `al->element_count` times, so for an empty literal `[]` the loop body never ran and `elem_type` stayed `NULL`. The fallthrough at line 1884 produced `iron_type_make_primitive(IRON_TYPE_ERROR)` and the literal resolved to `[<error>]`. The var-decl handler at line 2216+ later compared `[<error>]` against the declared `[Shape]` annotation and emitted the misleading `E0202: type mismatch expected '[Shape]', got '[<error>]'`.

The declared annotation was known at the var-decl handler BEFORE `check_expr` was called, but there was no mechanism to thread it into the ARRAY_LIT case. The fix threads the expected type via a thin `check_expr_with_expected` wrapper at the 4 expression contexts where an expected type is statically available (var decl, call arg, return, assign), keeping `check_expr`'s signature and all other ~20 callers untouched.

## Fix Approach

**1. Helper wrapper (~15 lines):**
```c
static Iron_Type *check_expr_with_expected(TypeCtx *ctx, Iron_Node *node,
                                            Iron_Type *expected) {
    if (node && node->kind == IRON_NODE_ARRAY_LIT && expected &&
        expected->kind == IRON_TYPE_ARRAY) {
        Iron_ArrayLit *al = (Iron_ArrayLit *)node;
        if (al->element_count == 0) {
            al->resolved_type = expected;
            return expected;
        }
    }
    return check_expr(ctx, node);
}
```

**2. 4 callsite swaps:**
- `typecheck.c:2270` (VAR_DECL): `check_expr(ctx, vd->init)` → `check_expr_with_expected(ctx, vd->init, decl_type)`
- `typecheck.c:1337-1338` (CALL arg loop): reordered so `param_type` is fetched first, then `check_expr(ctx, ce->args[i])` → `check_expr_with_expected(ctx, ce->args[i], param_type)`
- `typecheck.c:2379` (RETURN): `check_expr(ctx, rs->value)` → `check_expr_with_expected(ctx, rs->value, ctx->current_return_type)`
- `typecheck.c:2326` (ASSIGN): `check_expr(ctx, as->value)` → `check_expr_with_expected(ctx, as->value, target_type)`

**3. E0229 emission in ARRAY_LIT case:** when `element_count == 0 && !elem_type`, emit the targeted diagnostic before falling through to `IRON_TYPE_ERROR`. The helper short-circuits this path for callers that thread an expected type, so E0229 only fires for the no-context case.

## Deferred Issues

**Interface arrays as function parameters — pre-existing codegen gap (unrelated to Phase 55.1):**

When a function takes an interface array parameter (e.g., `func f(shapes: [Shape]) -> Int { return shapes.len() }`), the C emitter generates the function signature with a `Iron_SplitList_Iron_Shape` parameter type but uses regular `Iron_List_Iron_Shape_*` method calls inside the body, which are undeclared. The split-collection interception block in `src/lir/emit_c.c` handles `_push`/`_len`/`_pop`/`_get`/`_set` suffix detection for local variables of type `[Shape]`, but the method calls inside a function whose parameter is typed `[Shape]` are not routed through that interception path.

**Reproduction:**
```iron
interface Shape { func area() -> Int }
object Circle impl Shape { var radius: Int }
object Square impl Shape { var side: Int }
func Circle.area() -> Int { return self.radius * self.radius * 3 }
func Square.area() -> Int { return self.side * self.side }

func count_shapes(shapes: [Shape]) -> Int {
  return shapes.len()   -- codegen: Iron_List_Iron_Shape_len (undeclared)
}

func main() {
  var populated: [Shape] = [Circle(1), Square(2)]
  var n = count_shapes(populated)
  println("{n}")
}
```
Error: `error: call to undeclared function 'Iron_List_Iron_Shape_len'`

This gap affects any function parameter of interface array type and is unrelated to Phase 55.1. It should be tracked in `.planning/REQUIREMENTS.md` as a separate requirement for a future phase (candidate naming: `IFACE-PARAM-01`).

**Remediation hints for the future phase:**
- Hook the HIR→LIR lowering for method calls on function parameters whose type is `[<interface>]`
- Or: rewrite the interception block in `emit_c.c` to detect method calls on any variable with an interface-array type (not just locally-constructed split collections)
- Likely touches `src/hir/hir_to_lir.c` and the `strcmp(suffix, "_len") == 0` branch in `src/lir/emit_c.c`
