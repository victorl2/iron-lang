**hir_to_lir.c — Register monomorphized enum instances:**
After the existing enum pre-pass loop, add a NEW loop that scans for monomorphized enum types. These are created by the type checker in `resolve_type_annotation` and stored as `Iron_Type` with `enu.mangled_name != NULL`. We need to collect all such types and register them.

The approach: walk all resolved types in the program's declarations (function signatures, variable declarations) to find monomorphized enum types and register each unique one. Add a helper function and a collection loop:

```c
/* Collect monomorphized enum types by walking resolved types in all declarations */
/* Use a stb_ds string hash to track already-registered mangled names */
struct { char *key; bool value; } *mono_seen = NULL;

/* Walk function bodies to find monomorphized enum types.
 * We scan IRON_NODE_ENUM_CONSTRUCT nodes which have resolved_type set by the type checker. */
for (int i = 0; i < ctx->program->decl_count; i++) {
    Iron_Node *decl = ctx->program->decls[i];
    if (!decl) continue;
    /* Scan function bodies for enum constructs and variable types */
    collect_monomorphized_enums(ctx, decl, &mono_seen);
}
shfree(mono_seen);
```

Implement `collect_monomorphized_enums` as a recursive AST walker that:
1. For `IRON_NODE_ENUM_CONSTRUCT`: check `ec->resolved_type` — if it has `enu.mangled_name != NULL` and hasn't been seen, register it via `iron_lir_module_add_type_decl` with `td->name = type->enu.mangled_name`.
2. For `IRON_NODE_VAR_DECL` / `IRON_NODE_VAL_DECL`: check `resolved_type` — same logic.
3. For `IRON_NODE_FUNC_DECL`: walk params and body.
4. Recurse into child nodes (body, args, etc.).

**Alternative simpler approach (preferred):** Instead of a full AST walker, leverage the fact that the type checker already creates monomorphized types and stores them in `ec->resolved_type`. We can walk all enum constructs found during HIR lowering. Then expose this via a function or global. In the HIR pre-pass, iterate this collection and register each as a type_decl.

The cleanest integration point: After the type checker runs (`iron_typecheck` returns), the monomorphized types are stored inside the `Iron_Type` objects reachable from `ec->resolved_type` on ENUM_CONSTRUCT nodes and from variable types. **Implementation note:** A full recursive walker is verbose but straightforward. Focus on the key node types that can carry monomorphized enum types: ENUM_CONSTRUCT, VAR_DECL, VAL_DECL, FUNC_DECL (return type). Also check match arms since pattern types reference the enum.

The executor should implement a function like:
```c
static void register_mono_enum(HirLowerCtx *ctx,
                                struct { char *key; bool value; } **seen,
                                Iron_Type *type) {
    if (!type || type->kind != IRON_TYPE_ENUM) return;
    if (!type->enu.mangled_name) return;
    if (shgeti(*seen, type->enu.mangled_name) >= 0) return;
    shput(*seen, (char*)type->enu.mangled_name, true);
    iron_lir_module_add_type_decl(ctx->lir_module, IRON_LIR_TYPE_ENUM,
                                  type->enu.mangled_name, type);
}
```
And call it from a recursive walker that visits the AST nodes in function bodies.
