    Find where `ctx.includes` is populated (likely near the beginning of iron_codegen function). Add:
    ```c
    iron_strbuf_appendf(&ctx.includes, "#include \"runtime/iron_runtime.h\"\n");
    ```

    This ensures all generated .c files include the runtime header, giving access to Iron_String, Iron_print, Iron_Rc, collections, threading types, and all builtins.

    Also ensure the standard C includes are present. The codegen likely already emits `#include <stdint.h>`, `<stdbool.h>`, `<stdio.h>`, `<stdlib.h>`, `<string.h>`. Verify and add any missing ones needed by the runtime (e.g., `<stdatomic.h>` for rc, `<pthread.h>` for threading). Actually, the runtime header itself includes these, so the generated C just needs `#include "runtime/iron_runtime.h"`.

    BUT: the generated C is written to a temp file and compiled by clang with `-I` pointing to the source tree. The CLI (Plan 06) will handle the `-I` flag. For now, the codegen just emits the include directive.

    **gen_stmts.c** — Fix parallel-for chunk calculation:

    Replace lines ~279-280:
    ```c
    iron_strbuf_appendf(sb,
        "int64_t _chunk_size = (_total + 3) / 4; /* 4 chunks */\n");
    ```
    With:
    ```c
    codegen_indent(sb, ctx->indent + 1);
    iron_strbuf_appendf(sb,
        "int64_t _nthreads = Iron_pool_thread_count(Iron_global_pool);\n");
    codegen_indent(sb, ctx->indent + 1);
    iron_strbuf_appendf(sb,
        "int64_t _chunk_size = (_total + _nthreads - 1) / _nthreads;\n");
    ```

    This makes the chunk count dynamic based on the actual pool size at runtime.

    **gen_types.c** — Verify collection struct emission includes correct field names:
    Check that `ensure_monomorphized_type` emits structs with `items`, `count`, `capacity` fields matching the runtime collection macros. If the field names differ, update to match. The research says the codegen emits `{ T *items; int64_t count; int64_t capacity; }` which should match the runtime macros.

    Also verify: when the codegen emits `Iron_String` for IRON_TYPE_STRING, it doesn't need a forward declaration because the runtime header provides it. But the codegen's forward_decls might try to emit `typedef struct Iron_String Iron_String;` which would conflict. Check and skip Iron_String forward declaration if it's a runtime-provided type.
