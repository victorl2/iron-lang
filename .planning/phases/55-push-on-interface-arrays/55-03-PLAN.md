**Set semantics decision (Claude's Discretion, per CONTEXT.md line 45):** the simplest correct behavior for set is **same-type only**. If the new value's static type matches the current slot's type (as determined by `_order[i].tag`), write in place. If different, the simplest correct behavior is to emit a runtime assertion failure (via `fprintf(stderr, ...); exit(1);`) OR — even simpler — emit only the same-type write path guarded by a runtime tag check, and silently leave the collection unchanged on mismatch with a debug comment. The executor picks the approach that produces the cleanest C; either is acceptable.

**Actually simpler approach for set:** since `emit_get_value_type` on the pushed value returns the concrete static type at emit time, we know the type of `v` at compile time. So we can emit a runtime check `if (sl._order[i].tag == <expected_tag>) { sl.<lower>_items[sl._order[i].idx] = v; }` — this handles the common case (same-type overwrite) cleanly. For different-type, silently no-op OR assert. Document the choice in the commit message.

**Semantics summary (document in commit message):**
- Same-type set: in-place overwrite. Supported.
- Different-type set: silent no-op with a C comment. The value is not written, the collection is unchanged. This is the simplest correct behavior — it avoids reshuffling the `_order` array and the per-type sub-arrays, which would be non-trivial. Document as known limitation; alternative semantics can be revisited in a future phase if users request it.
- Interface-typed new value (e.g. `var s: Shape = Circle(5); shapes.set(0, s)`): not supported in Phase 55 — no-op with C comment. Same reasoning as the different-type case, plus the added complication of tag-switch dispatch on the write side.

**SoA constraint:** if the target type uses SoA storage, skip the write and emit a comment. Document.

**Note:** this approach is strictly simpler and safer than runtime assertions — no crashes, no reshuffle bugs. Users who hit the limitation will see the collection remain unchanged and can debug from there. If the executor prefers an assertion approach (emit `fprintf(stderr, "set: tag mismatch"); exit(1);` on mismatch), that is also acceptable; pick one and document.
