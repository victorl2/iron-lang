# Phase 55: Push on Interface Arrays - Context

**Gathered:** 2026-04-09
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix the broken `.push()` and other accessor/mutation method calls on interface-typed split collections (`[Shape]`). The root cause is already diagnosed: `src/lir/emit_c.c`'s split-collection interception block (lines 1501-1845) handles only the higher-order iteration methods (`_map`, `_filter`, `_reduce`, `_forEach`, `_sum`) and lets all other method calls fall through to generic direct-call emission, which prints a non-existent function name (e.g. `Iron_List_Iron_Shape_push`) verbatim. Phase 55 extends the interception block to cover the full accessor/mutation surface for split collections.

**Out of scope (belongs to other phases):**
- Monomorphic-collapsed method chain bug → Phase 56
- SoA + fusion Stor type mismatch → Phase 57
- Benchmark stabilization → Phase 58
- New collection methods beyond the existing stdlib API (e.g. `.first()`, `.sort()`) → future milestone

</domain>

<decisions>
## Implementation Decisions

### Scope breadth
- **Broad fix in one phase** — extend the `emit_c.c` interception block to cover `_push`, `_len`, `_pop`, `_get`, `_set` in a single coherent change. Same root cause, same fix pattern repeated per method. All five accessors are in scope.
- Rationale: reduces future debug surface, the unit of work is well-contained (one emit block), and Phase 54 already documented `.len()` failing with the identical pattern — shipping them together avoids a second "same-bug, different method" phase.

### Push argument dispatch
- **Ship both variants of `.push()` in Phase 55:**
  - **Concrete case:** `shapes.push(Circle(3))` — the pushed value has `IRON_TYPE_OBJECT` static type. Emit a direct call to `Iron_SplitList_<iface>_push_<TypeName>(&self, value)` by reading the arg's static type via `emit_get_value_type(fn, push_val)`. Mirrors the existing array-literal path at `src/lir/emit_c.c:2555-2588` and fusion terminal at `src/lir/emit_fusion.c:370-383`.
  - **Interface-typed case:** `var s: Shape = ...; shapes.push(s)` — the pushed value has interface (tagged union) type. Emit a runtime `switch` over `s.tag` dispatching to the matching per-type push. The tag union is already available at emit time.
- No compile error / TODO fallthrough. Both cases produce correct C.

### Len / pop / get / set semantics
- **`_len`:** Returns sum of per-type sub-array lengths. Trivial dispatch — no per-type fan-out, emit a single expression summing `self->sub_<Type>.len` across all alive implementors (or use the existing `_order` length if that's authoritative).
- **`_pop`:** Consult the `_order` array to find the last-pushed element's type tag, pop from that sub-array, shrink `_order`. Returns the popped element boxed back into the interface tagged union.
- **`_get(i)`:** Use `_order[i]` to look up `(tag, sub_index)`, read from the matching typed sub-array, return boxed as the interface tagged union.
- **`_set(i, v)`:** Use `_order[i]` to find the current slot. If `v` is the same concrete type as the slot, in-place write. If `v` is a different concrete type, the semantics are TBD during planning (likely: error at runtime, or reshuffle — planner decides, but prefer the simplest correct behavior).
- All five methods stay within the existing closed-world static-dispatch model. No new runtime code — piggyback on `Iron_SplitList_<Iface>` fields already generated by `src/lir/emit_split.c`.

### Dispatch implementation style
- **Inline per method** — each of push/len/pop/get/set gets its own branch inside the existing interception block in `src/lir/emit_c.c`. No shared helper extraction. Mirrors the existing map/filter/reduce/forEach/sum style: 5 branches of ~10-25 lines each, self-contained, readable side-by-side.
- Rationale: the pattern varies per method (push is arg-type-dispatched, len is a sum, pop/get/set go through `_order`). Premature abstraction would obscure the per-method logic. Inlining keeps each method's behavior local.

### Claude's Discretion
- Exact emitted C for each branch (variable naming, temporary handling, brace style) — follow existing emit_c.c conventions.
- How to handle the `_set` same-type vs. different-type case if the obvious answer isn't clean during implementation — flag to user if non-trivial.
- Whether tag-switch push falls through to a helper function emitted into the split-collection header, or is inlined at each call site. Prefer whichever matches the existing emit_c.c / emit_split.c split for similar concerns.
- Mangling details for the tag-switch dispatcher (if factored out).

</decisions>

## Existing Code Insights

### Reusable Assets
- **`emit_get_value_type(fn, value_id)`** — Returns the concrete `Iron_Type*` for a LIR value. Already used in the array-literal path and fusion terminal to pick per-type push at emit time. The Phase 55 `_push` branch uses it identically.
- **`Iron_SplitList_<Iface>_push_<Type>` functions** — Already generated in `src/lir/emit_split.c:398-611`. Self-contained: grow per-type sub-array, append, update `_order[]` with `(tag, idx)`, increment counts. Zero runtime changes required.
- **`Iron_SplitList_<Iface>` struct layout** — Per-type sub-arrays (`sub_<Type>`), `_order` array tracking push order with `(tag, sub_index)` pairs, total count. All Phase 55 methods operate on these existing fields.
- **Interface tagged union** — Available at emit time for interface-typed values. The runtime-dispatched push (case `var s: Shape = ...; shapes.push(s)`) switches on `s.tag` to pick the right per-type push.
- **`sp_iface`, `sp_entry`, `self_arg`** — Already in scope at the fix site in `emit_c.c`. The interception block already has everything needed; Phase 55 just adds new branches.

### Established Patterns
- **Interception-block dispatch** — Phase 47 established the pattern: frontend produces uniform `Iron_List_<elem>_<method>` names; the C emitter intercepts them for split collections and rewrites per-type. Phase 55 extends the same block with new method cases.
- **Per-type static dispatch** — All existing split-collection method emission (map, filter, reduce, forEach, sum, array literal, fusion terminal) dispatches by the static concrete type of the relevant value. Phase 55 follows the same pattern for `_push` with concrete arg; `_len`/`_pop`/`_get`/`_set` use `_order[]` to recover the type at the indexed slot.
- **Integration test style** — `tests/integration/*.iron` + `.expected` output file, auto-discovered. Verification combines runtime output match + `grep` of generated C to confirm the intended path was taken. Phase 55's regression tests should grep for `Iron_SplitList_Iron_Shape_push_Circle` / `_push_Square` to prove the fix landed.

### Integration Points
- **Phase 54 workarounds ready to remove** — `tests/integration/stress_large_collection.iron` uses typed Int push loop instead of the natural interface push because of PUSH-01. Planner should decide whether to remove that workaround as part of Phase 55 or leave as-is (it still exercises arena growth via typed array).
- **`emit_ctx_cleanup()` and the emitter context** — all per-phase emitter state already cleaned up by existing code; Phase 55 adds no new ctx fields.

<specifics>
## Specific Ideas

- The fix pattern for `_push` is already sketched in the debug file's `Resolution.fix` section (`.planning/debug/push-split-collection.md`) as a ready-to-adapt code skeleton. Planner should treat it as the starting point for the primary branch, then layer the tag-switch variant on top.
- Iron_SplitList invariant: `_order[i] = (tag, sub_index)` where `tag` indexes into the alive implementors and `sub_index` is the position within `sub_<Type>`. All five Phase 55 methods rely on this invariant staying intact.
- Commit message for the fix must include the root cause summary (per ROADMAP.md success criterion #3): what was broken (interception block missing accessor/mutation cases), why (Phase 47 scoped only to iteration methods), how fixed (added per-method branches mirroring the existing pattern).

</specifics>
