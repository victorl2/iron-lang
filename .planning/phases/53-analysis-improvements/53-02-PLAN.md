<objective>
Extend the monomorphic detection pre-scan in emit_c.c to track concrete types across function boundaries -- both through function return values and function parameters.

Purpose: Currently, monomorphic detection is local-only (only ARRAY_LIT within the same function). A helper function like `buildCircles()` that returns `[Circle]` does NOT trigger monomorphic collapse at the call site. This plan adds interprocedural tracking so that functions returning single-type collections and functions receiving single-type arguments can benefit from monomorphic optimization.

Output: Extended monomorphic pre-scan in emit_c.c with cross-function return tracking, parameter type tracking, and heuristic-gated specialization. Two integration tests.
</objective>

**Step 1: Add pre-pass to collect per-function return concrete types**
BEFORE the existing per-function monomorphic scan, add a new pre-pass ("Phase 0: Function return type collection"):
- Create a string hash map: `struct { char *key; const char **value; } *func_return_types = NULL;` where key = function name, value = stb_ds array of concrete type names found in RETURN instructions
- For each function in the module, scan all blocks for RETURN instructions
- For each non-void RETURN, check if the returned value is:
  - An ARRAY_LIT that is a split collection: extract concrete types from its elements (same logic as Phase A)
  - A value that traces through STORE/LOAD back to an ARRAY_LIT: track using vid_origin propagation
- Collect the set of concrete type names per function
- Store in func_return_types: `shput(func_return_types, fn->name, types_array)`

**Step 2: Use return types at CALL sites in the existing Phase C**
In the existing Phase C ("Check for escaping collections"), currently collections passed as CALL args are marked as escaped. Modify this logic:
- Instead of unconditionally escaping collections passed as CALL args, check if the CALL result is itself stored into a tracked split collection
- When a CALL's return value is stored into a split collection ValueId, AND the callee function has a known set of return types (from func_return_types), add those types to the coll_types for the receiving collection instead of escaping it

Additionally, for CALL instructions where the result is used as a collection (appears in split_collection_ids): look up func_return_types for the callee. If it has exactly 1 type, add the CALL result's ValueId to coll_types with that type set.

**Step 3: Handle RETURN of collections -- don't escape when tracked**
When a function RETURNs a collection, instead of marking it escaped, check if the collection's type set is tracked. If the collection's types are known (from coll_types), record them in func_return_types for this function. This feeds back into Step 2 for callers.

**Step 4: Clean up func_return_types**
After the pre-scan, free all the type arrays and the hash map.

After the existing monomorphic pre-scan (Phase D), add "Phase E: Parameter type collection":
- Create a hash map: `struct { char *key; const char **value; } *param_types = NULL;` where key = "func_name:param_index", value = stb_ds array of concrete type names passed at call sites
- Scan ALL CALL instructions across all functions in the module
- For each CALL argument that is a tracked split collection (in split_collection_ids) AND has known types (in coll_types or monomorphic_collections), record those types for the parameter position
- Key = "callee_name:param_idx" -> union of all concrete types seen

**Step 2: Apply specialization heuristic (per CONTEXT.md)**
For parameters where all call sites pass the same single concrete type, check specialization eligibility:
- Count the callee function's BRANCH_IF instructions that check collection element tags as "dispatch overhead" indicator
- Count total instructions as a size proxy
- Specialization heuristic: specialize ONLY when ALL of:
  1. The function has <= 50 LIR instructions (adjustable threshold)
  2. There are 1-2 call sites for this function
  3. The function body has at least 1 dispatch-related BRANCH (SWITCH on tag or BRANCH checking element type)
- When conditions met: mark the callee's parameter collection as monomorphic (hmput into monomorphic_collections using the parameter's internal ValueId in the callee function)
- When conditions NOT met: use union of all call-site types (conservative, no specialization). Log this as a diagnostic if --warn-fusion-break is enabled.

**Step 3: Leverage existing specialization_registry**
Use the existing `specialization_registry` from Phase 49 to prevent duplicate specialized function bodies when the same function is specialized for the same concrete type from multiple call sites.
