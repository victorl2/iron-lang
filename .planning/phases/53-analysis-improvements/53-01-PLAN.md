<objective>
Extend value range analysis with interprocedural return range propagation and conditional branch narrowing.

Purpose: Currently, CALL instruction results are conservative TOP (no compression possible) and conditional branches do not narrow variable ranges. This plan adds two analysis extensions to value_range.c that enable compression in more cases: (1) function return ranges propagate to call sites, and (2) conditional comparisons narrow ranges in branch targets.

Output: Extended value_range.h/c with return range propagation and conditional narrowing. Two integration tests proving the optimizations work end-to-end.
</objective>

    if (instr->call.func_decl && instr->call.func_decl->name && vra->func_return_ranges) {

    break;

Add conditional range narrowing to the per-function analysis in value_range.c. Per CONTEXT.md decisions: handle simple comparisons and AND chains; skip OR union merging.

**Step 1: Add per-block entry range map**
Create a new internal data structure for block-scoped ranges:
```c
typedef struct {
    IronLIR_BlockId key;
    VREntry *value;  /* per-value range overrides for this block's entry */
} BlockRangeEntry;
```
Add `BlockRangeEntry *block_entry_ranges = NULL;` local variable in `analyze_function_ranges`.

**Step 2: Detect narrowing at BRANCH terminators**
At the end of each block's instruction scan (after processing all instructions), check if the block's terminator is an IRON_LIR_BRANCH. If so:
- Find the condition value (branch.cond) in value_table
- Check if the condition is a comparison instruction (LT, LTE, GT, GTE, EQ, NEQ)
- Check if one operand is a CONST_INT and the other is a tracked value with a known range
- Compute narrowed ranges for true and false branches:
  - `x < C`: true branch gets [existing_min, C-1], false branch gets [C, existing_max]
  - `x <= C`: true branch gets [existing_min, C], false branch gets [C+1, existing_max]
  - `x > C`: true branch gets [C+1, existing_max], false branch gets [existing_min, C]
  - `x >= C`: true branch gets [C, existing_max], false branch gets [existing_min, C-1]
  - `x == C`: true branch gets [C, C], false branch keeps existing range
  - `x != C`: true branch keeps existing range, false branch gets [C, C] (only useful if existing was already [C, C])
- For the variable operand, store the narrowed range in block_entry_ranges for the target block IDs (then_block and else_block). Handle the case where the variable is accessed through LOAD/STORE chains by looking up the alloca_ranges map.

**Step 3: Apply block entry ranges at block start**
Before processing each block's instructions, check if block_entry_ranges has an entry for this block. If so, merge the narrowed ranges into the working value_ranges map. For AND chains: if a block already has entry ranges from a dominating block's narrowing, intersect (use range with tighter bounds: max of mins, min of maxes) rather than replace.

To implement AND chain accumulation: when recording narrowed ranges for a target block, if the target block already has entry ranges (from a prior dominating block's narrowing of the same variable), intersect the two ranges. This handles `&&` short-circuit blocks where block B is dominated by block A and both branch on the same variable.

**Step 4: Clean up**
Free block_entry_ranges and all nested VREntry maps at the end of analyze_function_ranges.
