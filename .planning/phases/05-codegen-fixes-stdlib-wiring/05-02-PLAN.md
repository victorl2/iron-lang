<objective>
Fix the parallel-for codegen so that chunk functions receive correct start/end values and can access captured outer variables.

Purpose: The current parallel-for codegen at gen_stmts.c:242 has a critical bug: it casts the loop start value `_c` to `void*` and passes it to `Iron_pool_submit`, but the chunk function has a 3-argument signature `(int64_t start, int64_t end, void* ctx_arg)` which is incompatible with `void(*)(void*)`. The start/end values are never actually passed to the chunk function, and outer variables referenced in the body are not captured. This plan fixes both issues.

Output: Working parallel-for that distributes work correctly across threads with proper capture of outer variables.
</objective>

**REVISED SIMPLEST APPROACH**: Instead of using collect_captures at all for captures, use a different strategy:

The chunk function body will reference outer variables. Since the chunk function is a top-level C function, those names won't be in scope. The solution: pass a POINTER to each captured variable through the context struct, and shadow the names with pointer dereferences.

But actually, for the canonical case `array[i] = i`, `array` is already a pointer (C arrays decay to pointers). So passing `array` as `void*` and casting back to `int64_t*` works perfectly.

For scalar captures (e.g., `val n = 10` used inside parallel for), pass `&n` and deref as `*(int64_t*)`.

Since we can't easily distinguish array-vs-scalar at codegen time without type info, use this heuristic: ALL captures are passed as `void*` in the struct. In the chunk function body, the codegen does NOT emit shadow variables -- instead, it emits the body normally, and relies on the captures being set up as local variables before the loop.

Actually, this is still complex. Let me take the absolute simplest approach that works for the success criteria:

**SIMPLEST CORRECT APPROACH**:
- Captures are stored as `void*` in the context struct
- At the use site, `&variable` is stored (address of the outer variable)
- In the chunk function, before the loop body, emit a local pointer variable for each capture: `int64_t *capname = (int64_t*)_pctx->cap_capname;`
- The body emission uses the original variable name. For `array[i] = i`, the codegen emits `capname[i] = i` which is... wrong, it would emit `array[i] = i` which uses the local pointer `array`.

Wait, that actually WORKS! If we declare `int64_t *array = (int64_t*)_pctx->cap_array;` at the top of the chunk function, then the body line `array[i] = i` will use this local `array` pointer, which points to the outer array. This is correct!

So the full implementation:

**Capture struct** (in lifted_funcs, before chunk function):
```c
typedef struct {
    int64_t start;
    int64_t end;
    void *cap_array;  // for each capture
} Iron_parallel_ctx_N;
```

**Chunk function** (in lifted_funcs):
```c
void Iron_parallel_chunk_N(void* ctx_arg) {
    Iron_parallel_ctx_N *_pctx = (Iron_parallel_ctx_N*)ctx_arg;
    int64_t _start = _pctx->start;
    int64_t _end = _pctx->end;
    int64_t *array = (int64_t*)_pctx->cap_array;
    for (int64_t i = _start; i < _end; i++) {
        array[i] = i;
    }
    free(ctx_arg);
}
```

**Use site** (in main body):
```c
{
    int64_t _total = <range_expr>;
    int64_t _nthreads = Iron_pool_thread_count(Iron_global_pool);
    int64_t _chunk_size = (_total + _nthreads - 1) / _nthreads;
    for (int64_t _c = 0; _c < _total; _c += _chunk_size) {
        int64_t _end = (_c + _chunk_size > _total) ? _total : _c + _chunk_size;
        Iron_parallel_ctx_N *_ctx = (Iron_parallel_ctx_N*)malloc(sizeof(Iron_parallel_ctx_N));
        _ctx->start = _c;
        _ctx->end = _end;
        _ctx->cap_array = (void*)array;
        Iron_pool_submit(Iron_global_pool, Iron_parallel_chunk_N, _ctx);
    }
    Iron_pool_barrier(Iron_global_pool);
}
```

Note: `free(ctx_arg)` at the END of the chunk function frees the per-chunk malloc.

**Prototype** (in prototypes buffer):
