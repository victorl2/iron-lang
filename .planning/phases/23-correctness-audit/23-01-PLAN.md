  <action>
Trace every HIR instruction kind through its full round-trip path: hir.h definition -> hir_to_lir.c lowering -> lir.h LIR instruction -> emit_c.c C emission. For each kind, verify the lowering produces the correct LIR instruction and the emission produces valid C code.

**Methodology:**

1. Read hir.h to enumerate all 13 IRON_HIR_STMT_* kinds and all 28 IRON_HIR_EXPR_* kinds (lines 27-83, lines 147-383 for payloads).

2. For each stmt kind, find its `case IRON_HIR_STMT_*:` in hir_to_lir.c lower_stmt(), trace what LIR instructions it produces, then find those LIR instructions in emit_c.c emit_instr() and verify the C output is correct.

3. For each expr kind, find its `case IRON_HIR_EXPR_*:` in hir_to_lir.c lower_expr(), trace what LIR instructions it produces, then find those LIR instructions in emit_c.c emit_instr() and verify the C output is correct.

4. Pay special attention to these HIGH/MEDIUM risk paths identified in research:
   - **EXPR_CLOSURE** (HIGH): Check the `(ret (*)())` cast in emit_c.c ~line 1439. Document whether it emits proper parameter types or uses the deprecated no-prototype cast. Note this as a known limitation if the cast still omits parameter types.
   - **GET_FIELD arrow vs dot** (MEDIUM): Check emit_c.c ~lines 1145-1150 where `obj_is_ptr` is determined by checking if the producing instruction was HEAP_ALLOC or RC_ALLOC. Verify this works correctly when a heap object is stored into an alloca and later loaded (the InstrKind would be LOAD, not HEAP_ALLOC). Cross-reference with the hir_edge_method_on_heap test to verify the path works.
   - **PARALLEL_FOR capture zero** (MEDIUM): Check hir_to_lir.c ~line 899-903 where `NULL, 0` is passed for captures. Document the current behavior: pfor bodies are pure functions with no outer-variable access. Note whether attempting to access outer variables produces a compile error or silent wrong behavior.
   - **EXPR_IS** (MEDIUM): Check hir_to_lir.c ~line 979 for the IS_CHECK lowering. Verify emit_c.c handles IS_CHECK correctly (tag comparison on interface vtable). Note if there is a dedicated test or if this is untested.
   - **EXPR_METHOD_CALL** (MEDIUM): Check the self-alloca + CALL pattern. Verify that method calls on heap objects correctly dereference the pointer (arrow vs dot for self parameter).

5. For each instruction kind, use `--verbose` flag on a representative test to inspect actual generated C where needed:
   ```bash
   ./build/ironc build tests/integration/TESTNAME.iron -o /tmp/audit_test --verbose 2>&1 | head -100
   ```

6. Write the checklist to `.planning/phases/23-correctness-audit/audit-checklist.md` with the following format for each entry:
   ```
   - [x] KIND_NAME: [brief description of what was checked] — CHECKED [OK | ISSUE: description]
   ```
   Use `CHECKED OK` for kinds that are correct. Use `CHECKED ISSUE: ...` for any bug found.

**What to look for (bug indicators):**
- LIR instruction missing a field that the HIR node carries (data loss in lowering)
- emit_c.c case that uses wrong accessor (e.g., `instr->call` when it should be `instr->method_call`)
- Missing NULL checks on optional fields (e.g., optional initializer in STMT_LET)
- Incorrect pointer/value distinction (arrow vs dot in generated C)
- Uncasted void* in arithmetic context
- `Iron_` prefix on raw type in generated C

**If bugs are found:** For each bug, create a minimal `.iron` test that triggers the bug, compile it to capture the current (wrong) output, then fix the code in hir_to_lir.c or emit_c.c, recompile the test to get the correct output, and save both .iron and .expected files as regression tests. Name regression tests `tests/integration/bug_audit_DESCRIPTION.iron`.
