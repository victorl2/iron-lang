<objective>
Implement monomorphic collection collapse and specialization registry. Collections proven to hold only one concrete type collapse to plain typed arrays with direct field access, eliminating all split collection overhead.

Purpose: When whole-program analysis proves a collection only contains one concrete type, eliminate the tagged union dispatch, split sub-arrays, and order array — emit a simple `Iron_List_<ConcreteType>` instead. The specialization registry prevents duplicate function body emission.

Output: Monomorphic detection pass in emit_c.c, collapse emission, specialization registry, 3 integration tests.

    **Implement three components: monomorphic detection, collapse emission, and specialization registry.**

    **1. Add EmitCtx fields** (after the Phase 49 fusion fields from Plan 01):
    ```c
    /* Phase 49: Monomorphic collection tracking */
    struct { IronLIR_ValueId key; const char *value; } *monomorphic_collections;
        /* maps collection ValueId -> sole concrete type name (e.g., "Circle") */
        /* only populated when a split collection has exactly one type pushed */

    /* Phase 49: Specialization registry */
    struct { char *key; const char *value; } *specialization_registry;
        /* maps "func_name:concrete_type" -> emitted C function name */
        /* prevents duplicate function body emission for same specialization */
    ```

    **2. Monomorphic detection pass** — add as a whole-program pre-scan that runs BEFORE per-function emission. Place it in the `iron_lir_emit_c()` main function (around line ~6020, before the loop that calls `emit_func_body` for each function). This is a module-level analysis, not per-function.

    **Algorithm:**
    a. Create a tracking map: `struct { IronLIR_ValueId key; struct { const char **types; int count; } value; } *coll_types = NULL;`
       This maps collection ValueIds to the set of concrete types pushed to them.

    b. Scan ALL functions in the module. For each function, scan all blocks:
       - Find ARRAY_LIT instructions with interface elem type. These are collection creation sites. Record the ValueId.
       - Propagate collection identity through STORE/LOAD chains (same pattern as split_collection_ids propagation).
       - Find CALL instructions that are push operations or CONSTRUCT-into-collection patterns. When a CONSTRUCT instruction's result is pushed (via method call) to a tracked collection, record the construct's type_name.
       - Alternatively, find CONSTRUCT instructions whose result flows into an ARRAY_LIT's element list. Record those type names.

    c. For interface-typed arrays, also check the ARRAY_LIT's elem list: each element is typically a CONSTRUCT or value that can be traced to a concrete type. Check `fn->value_table[elem_vid]` — if it's a CONSTRUCT, record its type_name.

    d. After scanning all functions, for each tracked collection:
       - If exactly 1 unique concrete type was found → mark as monomorphic
       - Store in `ctx->monomorphic_collections` (ValueId -> type_name)

    **Conservative approach per RESEARCH.md Open Question 3**: Only detect collections created and fully used within the same function. Do NOT track collections passed as parameters (may receive different types from different callers). Specifically:
    - Only consider ARRAY_LIT instructions, not parameter ValueIds
    - Only consider CONSTRUCT elements in the same function's scope
    - If a collection escapes (passed to another function as argument), do NOT mark it monomorphic

    **3. Collapse emission** — modify the split collection detection in `emit_func_body` pre-scan:

    After the existing split collection detection builds `ctx->split_collection_ids`, add a second pass:
    ```c
    /* Phase 49: Monomorphic collapse — if a split collection has only one
     * concrete type, replace it with a plain typed array. */
    if (ctx->monomorphic_collections) {
        IronLIR_ValueId *to_unsplit = NULL;
        for (ptrdiff_t i = 0; i < hmlen(ctx->split_collection_ids); i++) {
            IronLIR_ValueId vid = ctx->split_collection_ids[i].key;
            ptrdiff_t mi = hmgeti(ctx->monomorphic_collections, vid);
            if (mi >= 0) {
                arrput(to_unsplit, vid);
            }
        }
        for (int i = 0; i < (int)arrlen(to_unsplit); i++) {
            hmdel(ctx->split_collection_ids, to_unsplit[i]);
        }
        arrfree(to_unsplit);
    }
    ```
    By removing the collection from `split_collection_ids`, the existing emission code will NOT emit `Iron_SplitList_<Iface>` — instead it must emit `Iron_List_<ConcreteType>`.

    Also, at the ARRAY_LIT emission site (where split collections are emitted as `Iron_SplitList_<Iface>`), add a check:
    ```c
    /* Phase 49: If this collection is monomorphic, emit as plain typed array */
    if (ctx->monomorphic_collections) {
        ptrdiff_t mi = hmgeti(ctx->monomorphic_collections, instr->id);
        if (mi >= 0) {
            const char *concrete_type = ctx->monomorphic_collections[mi].value;
            /* Emit as Iron_List_Iron_<ConcreteType> instead of Iron_SplitList_<Iface> */
            /* Each element: extract the concrete type from the tagged union wrapper
             * and push directly. */
        }
    }
    ```

    For the collapsed array:
    - Emit type as `Iron_List_Iron_{ConcreteType}` (e.g., `Iron_List_Iron_Circle`)
    - Initialize with `Iron_List_Iron_Circle_create()`
    - For each element in the ARRAY_LIT, the element is a CONSTRUCT wrapped in a tagged union. Extract the concrete value: instead of `Iron_Shape_from_Circle(circle_val)`, just push `circle_val` directly.
    - For iteration (for-loops over the collection), emit direct `items[i]` access with `Iron_{ConcreteType}` type (no tag dispatch, no switch, no _order array).
    - For method calls on the collection, emit direct `Iron_List_Iron_{ConcreteType}_{method}()` calls (no split dispatch).

    **4. Specialization registry** — prevents duplicate specialized function emission:

    Add to EmitCtx initialization (in `iron_lir_emit_c`, around line 5960):
    ```c
    ctx.specialization_registry = NULL;
    ```

    When emitting a specialized function body (a function that operates on a concrete type from a monomorphic collection), check the registry first:
    ```c
    char spec_key[512];
    snprintf(spec_key, sizeof(spec_key), "%s:%s", func_name, concrete_type);
    if (shgeti(ctx->specialization_registry, spec_key) >= 0) {
        /* Already emitted — reuse existing function name */
        const char *existing = shget(ctx->specialization_registry, spec_key);
        /* ... emit call to existing ... */
    } else {
        /* Emit specialized function and register */
        shput(ctx->specialization_registry, spec_key, specialized_name);
        /* ... emit the specialized function body ... */
    }
    ```

    The registry uses the same stb_ds `shput`/`shgeti` pattern as the existing `mono_registry` and `reduced_storage_types` maps.

    **5. Cleanup** — free the maps at the end of `iron_lir_emit_c()`:
    ```c
    hmfree(ctx.monomorphic_collections);
    shfree(ctx.specialization_registry);
    ```

    **IMPORTANT edge cases:**
    - Empty collections (no elements pushed) should NOT be considered monomorphic — they could receive elements later
    - Collections where elements are pushed conditionally (inside if/else) should track all possible types from all branches
    - The existing split collection emission path must remain untouched for non-monomorphic collections
