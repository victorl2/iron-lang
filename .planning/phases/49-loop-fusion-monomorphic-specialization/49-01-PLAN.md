    **Algorithm — implement these steps:**

    **Step A: Collect all fusible CALL instructions.**
    Scan all blocks/instructions. For each CALL instruction, resolve the func_ptr to a FUNC_REF name. If the name matches `Iron_List_*_{map,filter,reduce,forEach,sum}`, record it as a fusible call candidate. Store in a temporary stb_ds array:
    ```c

        IronLIR_ValueId call_vid;
        const char *method;        /* "map"/"filter"/"reduce"/"forEach"/"sum" */
        IronLIR_ValueId self_arg;  /* first argument (the collection) */
        IronLIR_ValueId *lambda_args;
        int lambda_arg_count;
        IronLIR_ValueId init_arg;  /* for reduce: second arg (init value) */
    } FusibleCallInfo;
    ```
    Use the same name-matching logic already at line 1909-1917 (strncmp "Iron_List_" + strrchr '_').

    **Step B: Build result-to-call reverse map.**
    Create `struct { IronLIR_ValueId key; int value; } *result_to_call = NULL;` mapping each fusible call's result_vid (== call_vid) to its index in the candidates array.

    **Step C: Propagate through STORE/LOAD chains.**
    Create a value-identity map `struct { IronLIR_ValueId key; IronLIR_ValueId value; } *val_origin = NULL;` that maps each ValueId back to the fusible call result it originated from (if any). Seed with all fusible call_vids mapping to themselves. Then scan all blocks: for STORE instructions where store.value is in val_origin, propagate to store.ptr. For LOAD instructions where load.ptr is in val_origin, propagate to the load's result id.

    **Step D: Build chains by linking calls.**
    For each fusible call candidate, check if its self_arg traces back to another fusible call's result (via val_origin). If yes, link them. Build chains by walking these links. A chain starts from a call whose self_arg does NOT trace to another fusible call.

    **Step E: Use-count escape check.**
    For each intermediate value in a chain (all nodes except the terminal), check the use_counts map (already computed at line 4136). If use_count > 1 for an intermediate call's result vid, break the chain at that point — the intermediate result is used elsewhere and cannot be eliminated. IMPORTANT: also check the STORE'd values — if the intermediate was stored to an alloca and the alloca's load count > 1, break the chain.

    **Step F: Filter out single-node "chains" and store results.**
    Discard any chain with only 1 node (nothing to fuse). For valid chains (2+ nodes):
    - Append to `ctx->fusion_chains` (stb_ds array)
    - For each node in the chain, store in `ctx->fusion_chain_member` (call_vid -> chain_index)
    - Store position in `ctx->fusion_chain_position` (call_vid -> position, where last position = terminal)
    - Check if the chain's source collection is in `ctx->split_collection_ids` — set `chain.is_split` and `chain.sp_iface` accordingly

    **Step F2: Emit --warn-fusion-break diagnostics.**
    After chain detection completes, if `ctx->warn_fusion_break` is true, scan the fusible call candidates for break points. A break point exists when:

    - An intermediate value has use_count > 1 (escape) — report: `"fusion chain broken: intermediate result of .{method}() used elsewhere"`

    Emit these as notes/warnings using `fprintf(stderr, "note: --warn-fusion-break: %s at %s:%d\n", message, source_file, line)`. Use the instruction's span information if available, or the function name as context. Only emit when `ctx->warn_fusion_break == true`.

    **Title:** `# @fusible Annotation Specification`

    **Sections:**

    1. **Overview** — The `@fusible` annotation marks collection methods as eligible for loop fusion optimization. When chained calls to fusible methods are detected, the compiler composes their per-element bodies into a single fused loop with no intermediate allocations.

    2. **Syntax** — `@fusible` appears on the line immediately before a `func` declaration:
       ```iron
       @fusible
       func [T].map[U](f: func(T) -> U) -> [U] {}
       ```
       Only valid before `func` and method declarations. A diagnostic is emitted if `@` is followed by anything other than `fusible`, or if `@fusible` is not followed by `func`.

    3. **Safety Requirements** — A method is safe to annotate `@fusible` if and only if:
       - It iterates over `self` (the collection) exactly once, processing each element independently
       - It applies a per-element transformation, predicate, or accumulation with no inter-element dependencies
       - It does not modify the source collection (no mutation of `self`)
       - It produces either a new collection (map, filter) or a scalar result (reduce, sum, forEach returns void)
       - The per-element operation is expressible as: element-in -> (transform | predicate-test | accumulate) -> element-out or scalar-update

    4. **Initial Fusible Methods** — List the 5 methods with their semantics:
       | Method | Category | Per-element operation | Fused behavior |
       |--------|----------|----------------------|----------------|
       | `map` | Transform | Apply `f(elem)` -> new elem | Inline transform, pass to next stage |
       | `filter` | Predicate | Apply `f(elem)` -> Bool | Conditional: if true, pass to next stage |
       | `reduce` | Accumulate | Apply `f(acc, elem)` -> new acc | Update accumulator inline |
       | `forEach` | Terminal | Apply `f(elem)` (side effect) | Execute body inline, no result |
       | `sum` | Terminal | `acc += elem` | Accumulate inline |

    5. **Fusion Chain Rules** — How chains are detected and composed:
       - A fusion chain is a sequence of calls where each call's result is used ONLY as the input to the next call
       - Chains break at non-fusible method calls, producing independent sub-chains
       - Intermediate results with use_count > 1 (used by something other than the next chain step) break the chain
       - Terminal operations (reduce, sum, forEach) must be the last operation in a chain
       - Map and filter can appear anywhere in a chain; reduce/sum/forEach only at the end

    6. **Per-Element Body Extraction** — How the compiler interprets each fusible method in a fused loop:
       - `map(f)`: `result = f(env, elem)` — the mapped value becomes the element for the next stage
       - `filter(f)`: `if (!f(env, elem)) continue;` — skip element if predicate fails
       - `reduce(init, f)`: `acc = f(env, acc, elem)` — accumulate, initialized once before the loop
       - `forEach(f)`: `f(env, elem)` — execute, no result propagation
       - `sum()`: `acc += elem` — accumulate with addition
       The compiler does NOT parse the Iron function body. It uses hardcoded knowledge of these method semantics. The `@fusible` annotation serves as a whitelist gate — only annotated methods participate in fusion.

    7. **Split Collection Behavior** — When a fused chain operates on an interface-typed split collection:
       - One fused loop per concrete type, iterating directly over that type's sub-array
       - For reduce/sum terminals, accumulation is sequential: type 1 loop first, then type 2 loop continues from the same accumulator
       - For map/filter terminals that produce a collection, each per-type loop pushes to its own sub-array in the result split collection

    8. **SoA-Aware Fusion** — When a split collection uses SoA layout (Phase 48):
       - Fused loops access per-field arrays directly instead of struct items
       - Struct reconstruction happens only when the lambda expects the full object

    9. **Compiler Flag** — `--warn-fusion-break`: when enabled, the compiler emits a diagnostic at each point where a chain is broken by a non-fusible call, helping developers understand which operations prevent fusion. Usage:
       ```
       ironc build --warn-fusion-break myfile.iron -o myfile

       note: --warn-fusion-break: fusion chain broken: non-fusible call after .map() in function main
       note: --warn-fusion-break: fusion chain broken: intermediate result of .map() used elsewhere in function main
       ```
