<!-- The helper consumed at every call-site (new from 67-02) -->

From src/diagnostics/diagnostics.h:
```c
__attribute__((noreturn))
void iron_oom_abort(const char *where);
```

<!-- The two idioms used in this plan -->

Idiom A — iron_oom_abort guard (default):
```c
Iron_BinaryExpr *e = ARENA_ALLOC(p->arena, Iron_BinaryExpr);
if (!e) iron_oom_abort("parser.c:iron_parse_binop");
e->span = ...;
```

Idiom B — SAFETY annotation (only when a clear invariant protects the call):
```c
/* SAFETY: parent caller (iron_parse_program) pre-allocates the top-level
 * decl array below its arena reservation headroom; this alloc cannot
 * fail on any input the lexer can produce. */
Iron_Program *prog = ARENA_ALLOC(p->arena, Iron_Program);
```

**Decision rule**: Use Idiom A by default. Use Idiom B only when:
1. The alloc is bounded by a compile-time constant (e.g., `ARENA_ALLOC` for a fixed-size wrapper struct at the top of a function), AND
2. The parser has a backup failure path (e.g., returns NULL which the caller already checks)

For stretches of 10+ consecutive allocations in one function, consider a single leading comment `/* SAFETY: bounded by <N> call-sites for function scope; see preceding guard. */` and add an iron_oom_abort guard at the FIRST alloc of the function as the canary. This keeps the idiomatic weight low while still producing a clear OOM path.

**Step 3: Walkthrough — top-to-bottom**

Walk the file top to bottom. For every call to `iron_arena_alloc`, `ARENA_ALLOC`, or `iron_arena_strdup`, classify the call-site and apply the right idiom:

**Classification A — GUARD with iron_oom_abort (default case, expect ~90% of sites):**

Immediately after the assignment line, insert:
```c
if (!<var>) iron_oom_abort("parser.c:<function_name>");
```

Where `<function_name>` is the enclosing function (grep backward for `^[a-zA-Z_].*\(.*\).*{` to find the enclosing `Iron_<Kind> *iron_parse_<thing>(...)` signature). Use the function name verbatim in the string literal so OOM stderr output bisects to the exact call site.

Example rewrite of a representative site:
```c
/* Before */
Iron_BinaryExpr *e = ARENA_ALLOC(p->arena, Iron_BinaryExpr);
e->span = iron_span_merge(...);
e->kind = IRON_NODE_BINARY;

/* After */
Iron_BinaryExpr *e = ARENA_ALLOC(p->arena, Iron_BinaryExpr);
if (!e) iron_oom_abort("parser.c:iron_parse_binop");
e->span = iron_span_merge(...);
e->kind = IRON_NODE_BINARY;
```

For iron_arena_strdup sites:
```c
/* Before */
v->name = iron_arena_strdup(p->arena, tok->value, strlen(tok->value));

/* After */
v->name = iron_arena_strdup(p->arena, tok->value, strlen(tok->value));
if (!v->name) iron_oom_abort("parser.c:iron_parse_ident_strdup");
```

**Classification B — SAFETY annotation (rare, expect < 10% of sites):**

Only apply when there's a clear bounded-input invariant. Example:
```c
/* SAFETY: Iron_Program header alloc fires once per parser_create, bounded
 * by the maximum program size the lexer can produce (≤ INT_MAX tokens).
 * The arena is freshly created in iron_parser_create with >= 256 KB headroom,
 * so this allocation cannot fail for any conceivable input. */
Iron_Program *prog = ARENA_ALLOC(p->arena, Iron_Program);
```

The comment MUST cite a specific invariant — not "it's unlikely" or "it's fine". If you cannot state a concrete invariant, use Classification A instead. When in doubt, guard.

**Step 4: Avoid redundant guards in loops**

For loops that allocate N times (e.g., `for (int i = 0; i < n; i++) { ARENA_ALLOC(...) }`), add the guard inside the loop body. Do NOT hoist it outside — an OOM in the 50th iteration is as much a user-visible failure as OOM in the 1st.

**Step 5: Avoid redundant guards in comma-expression or trailing-assignment idioms**

Some Iron parser code may use `p->foo = ARENA_ALLOC(...);` followed immediately by field sets. Apply the guard after the assignment line:
```c
p->foo = ARENA_ALLOC(...);
if (!p->foo) iron_oom_abort("parser.c:<func>");
p->foo->bar = ...;
```
