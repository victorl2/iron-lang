**1. Update `spawn_independent_work/main.iron` to use await.**

This is the primary benchmark fairness fix. The benchmark has 8 direct spawns (not inside a loop) that do heavy computation. Change from fire-and-forget to handle+await pattern.

Replace the 8 spawn statements (lines 23-46) with:
```iron
    val h0 = spawn("task0") {
        var r = heavy_compute(0, 5000000)
    }
    val h1 = spawn("task1") {
        var r = heavy_compute(7, 5000000)
    }
    val h2 = spawn("task2") {
        var r = heavy_compute(13, 5000000)
    }
    val h3 = spawn("task3") {
        var r = heavy_compute(29, 5000000)
    }
    val h4 = spawn("task4") {
        var r = heavy_compute(37, 5000000)
    }
    val h5 = spawn("task5") {
        var r = heavy_compute(53, 5000000)
    }
    val h6 = spawn("task6") {
        var r = heavy_compute(67, 5000000)
    }
    val h7 = spawn("task7") {
        var r = heavy_compute(97, 5000000)
    }
```

Then after `val main_result = heavy_compute(101, 5000000)` and before `val elapsed = Time.now_ms() - start`, add awaits:
```iron
    -- Wait for all spawned tasks to complete (fair timing vs C pthread_join)
    await h0
    await h1
    await h2
    await h3
    await h4
    await h5
    await h6
    await h7
    val elapsed = Time.now_ms() - start
```

Update `expected_output.txt` if needed. The expected output should remain the same (it prints sequential verification results from the main thread; the awaited tasks don't contribute to output). Verify by reading the current expected output.

**2. Keep other spawn benchmarks as-is (fire-and-forget in loops is intentional).**

The following benchmarks have spawns inside `for iter in range(iters)` loops:
- `concurrency_spawn_captured` -- spawns inside loop, correctness test
- `concurrency_spawn_independence` -- spawns inside loop, correctness test
- `concurrency_spawn_result` -- spawns inside loop, correctness test
- `concurrency_pipeline` -- spawns inside loop, correctness test

These benchmarks can't easily capture handles inside a loop (no array-of-handles). The fire-and-forget pattern is intentional for testing spawn submission overhead. Leave the Iron source unchanged. They will emit compiler warnings (un-awaited spawn handle), which is acceptable.

**3. Keep `spawn_pipeline_stages` as-is (background workers have no downstream data dependency).**

Per RESEARCH.md recommendation: "Don't await background workers that have no downstream data dependency. Only await spawns that feed results used later." The bg_worker1/bg_worker2 spawns are purely background work; their results are never used. The C benchmark doesn't include them in timing either. Leave as fire-and-forget.
